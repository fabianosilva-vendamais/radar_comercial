01 Visão geral

O que precisa ser construído para sair do protótipo (sinais "a verificar") para sinais validados com fonte real.

O Radar Comercial monitora empresas-alvo (hoje importadas de planilha) e, para cada uma, deve cruzar sinais externos públicos com a memória comercial e gerar priorização + abordagem. O protótipo atual roda 100% no navegador: a IA (Claude Haiku) já produz a análise e as buscas a fazer, mas não executa a varredura da web — por limitação de CORS, ausência de servidor e de chaves de API seguras.

Este documento especifica a camada de backend que executa o Agente 3 (Radar externo) de verdade: consulta as fontes, normaliza os sinais, registra a URL da fonte + data de captura e devolve ao app sinais com status verificado em vez de a_verificar.

◆
Princípio inegociável (veracidade): todo sinal persistido deve carregar fonte, url e capturado_em. Nada entra como "fato" sem origem rastreável. Sinais inferidos por IA ficam como a_verificar até confirmação por fonte.

02 Arquitetura proposta

Frontend (este app) ↔ Backend (API + workers) ↔ Fontes externas ↔ Banco.

CLIENTE
App Radar (web)
Importa planilha, dispara rastreio, exibe sinais validados e abordagem.
API
Backend / Gateway
REST + autenticação. Recebe pedidos do app, enfileira jobs, serve dados.
WORKERS
Agentes (fila + cron)
Agentes 1–6 como jobs assíncronos. Rate-limit por fonte.
FONTES
APIs externas
Notícias, vagas, LinkedIn, CNPJ, IA. Cada uma com adapter próprio.
DADOS
PostgreSQL + cache
Empresas, contatos, sinais, rastreios, validações, auditoria.

O frontend nunca chama as APIs externas diretamente (chaves expostas + CORS). Ele fala só com o seu backend. O backend guarda as chaves, respeita os limites de cada fonte, normaliza e deduplica, e expõe um contrato estável (seção 7).

Por que assíncrono (fila + worker)

Rastrear 67+ contas em várias fontes leva tempo e esbarra em rate-limits. O fluxo correto: o app dispara POST /rastreios → backend cria um job e responde na hora com um job_id → o app acompanha o progresso por GET /rastreios/:id (ou WebSocket/SSE) e alimenta o medidor de progresso que já existe na tela.

06 Modelo de dados

Esquema relacional mínimo (PostgreSQL). O elo é sempre a empresa.

-- empresas: o elo central
CREATE TABLE empresas (
  id            uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  nome          text NOT NULL,
  cnpj          text,                 -- resolvido pelo Agente 1
  segmento      text, cidade text, uf text,
  site          text, linkedin text, porte text,
  score         int DEFAULT 0,         -- 0-100, calc. Agente 4
  origem        text,                 -- planilha | manual | api
  proveniencia  jsonb,                -- {tipo, arquivo, linha}
  validacao     text DEFAULT 'pendente', -- pendente|validado|incorreto
  ultimo_rastreio timestamptz,
  criado_em     timestamptz DEFAULT now(),
  UNIQUE(nome, uf)              -- upsert por nome+UF (não duplica)
);

-- contatos da conta
CREATE TABLE contatos (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  empresa_id uuid REFERENCES empresas(id) ON DELETE CASCADE,
  nome text, cargo text, email text, telefone text
);

-- sinais: SEMPRE com fonte rastreável
CREATE TABLE sinais (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  empresa_id uuid REFERENCES empresas(id) ON DELETE CASCADE,
  tipo text,            -- expansao|crescimento_comercial|investimento|...
  titulo text,
  resumo text,
  fonte text,           -- 'Google News' | 'BrasilAPI' | 'Proxycurl'...
  url text,             -- link da fonte original (obrigatório p/ verificado)
  relevancia int,       -- 0-100
  status text DEFAULT 'a_verificar', -- a_verificar|verificado|descartado
  hash text UNIQUE,     -- dedupe (empresa+tipo+url normalizada)
  capturado_em timestamptz DEFAULT now()
);

-- rastreios: cada execução do Agente 3 (job)
CREATE TABLE rastreios (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  escopo text,          -- 'empresa' | 'todas'
  empresa_id uuid,
  status text,          -- na_fila|rodando|concluido|erro
  progresso int DEFAULT 0, -- 0-100 p/ o medidor
  total int, processadas int,
  iniciado_em timestamptz DEFAULT now(), terminado_em timestamptz
);

-- auditoria de validação humana
CREATE TABLE validacoes (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  alvo text, alvo_id uuid,  -- 'empresa' | 'sinal'
  status text, usuario text, em timestamptz DEFAULT now()
);

07 Contrato de API (app ↔ backend)

Endpoints REST que o frontend já está pronto para consumir (espelham os objetos do protótipo).

POST   /v1/empresas/import      -- envia linhas da planilha → upsert por nome+UF
GET    /v1/empresas             -- lista paginada (?page=&q=&origem=&validacao=)
GET    /v1/empresas/:id         -- ficha + contato + sinais + memória
PATCH  /v1/empresas/:id         -- validar/marcar incorreta
DELETE /v1/empresas/:id

POST   /v1/rastreios            -- body {escopo:'todas'|'empresa', empresa_id?}
                                -- responde {job_id} na hora (assíncrono)
GET    /v1/rastreios/:id        -- {status, progresso, processadas, total}
                                -- (ou SSE/WebSocket p/ o medidor em tempo real)

PATCH  /v1/sinais/:id           -- {status:'verificado'|'descartado'}
GET    /v1/export/radar         -- briefing do dia (txt/json)
GET    /v1/export/empresa/:id   -- abordagem p/ o vendedor

08 Pipeline do Agente 3 (coletor)

Pseudo-código do worker que roda por empresa.

async function rastrearEmpresa(empresa, rastreio) {
  const sinais = [];

  // 1. Garante CNPJ (Agente 1) se faltar
  if (!empresa.cnpj) empresa.cnpj = await resolverCnpj(empresa.nome, empresa.uf);

  // 2. Consulta cada fonte em paralelo (com timeout + rate-limit)
  const [cadastral, noticias, vagas, pessoas] = await Promise.allSettled([
    brasilApi.alteracoes(empresa.cnpj),          // filiais, capital, sócios
    news.buscar(empresa.nome, { lang:'pt', country:'br' }),
    jobs.buscar(empresa.nome, ['SDR','vendedor','gerente comercial']),
    linkedin.mudancasExecutivas(empresa.nome),
  ]);

  // 3. Normaliza → {tipo,titulo,resumo,fonte,url,relevancia}
  sinais.push(...normalizar(cadastral, noticias, vagas, pessoas));

  // 4. Dedupe por hash(empresa+tipo+url) e filtra ruído
  const novos = dedupe(sinais).filter(s => s.relevancia >= 40);

  // 5. Persiste com status 'verificado' (têm url de fonte)
  await db.sinais.upsert(empresa.id, novos);

  // 6. Agente 4 (score) + Agente 5 (abordagem) via LLM
  const intel = await llm.analisar(empresa, novos);   // score + intenção
  const abordagem = await llm.estrategia(empresa, novos, intel);
  await db.empresas.update(empresa.id, { score: intel.score, ...abordagem });

  await rastreio.incrementar();  // alimenta o medidor de progresso
}
◆
Agende o Agente 6 (priorizador) como cron diário (ex.: 6h) para montar o ranking sem o usuário pedir — entregando o "quem abordar hoje" automaticamente.

09 Veracidade & verificação

  • Toda linha em sinais exige url para virar verificado. Sem URL → fica a_verificar (ex.: inferência de IA sem citação).
  • Guarde capturado_em e a resposta bruta da fonte (em um bucket/JSON) para auditoria e reprocessamento.
  • hash único evita o mesmo sinal entrar duas vezes em rastreios seguidos.
  • Use a data da fonte (publicação da notícia / data da vaga) para descartar sinal velho e calcular recência na relevância.
  • Quando usar Claude Web Search, persista as citações retornadas (cada uma já vem com URL) como a fonte do sinal.

10 Segurança & LGPD

  • Chaves de API só no servidor (variáveis de ambiente / secret manager). Nunca no frontend.
  • LGPD: os contatos (nome, e-mail, telefone) são dados pessoais. Defina base legal (legítimo interesse para prospecção B2B), permita exclusão, e registre origem do dado.
  • Termos de uso das fontes: respeite ToS e robots.txt. Para LinkedIn, use provedores autorizados — não faça scraping direto.
  • Rate-limit & cache: respeite limites por fonte; cacheie cadastral (muda pouco) por dias, notícias/vagas por horas.
  • Autenticação do app (login por usuário/equipe) + escopo de dados por organização.

14 Riscos & limitações

  • LinkedIn: a fonte mais valiosa é a mais restrita. Dependa de provedores autorizados; custo e cobertura variam.
  • Boards BR (Gupy/Catho): sem API oficial — use agregador (Google Jobs) para não violar ToS.
  • Falsos positivos: homônimos de empresa geram sinal errado. Por isso o CNPJ (chave única) e o desambiguador (site/LinkedIn) já existentes no app são essenciais.
  • Custo cresce com volume: 67 contas × várias fontes × frequência. Cacheie e priorize (rastreie diariamente só as contas quentes).
  • Bing Search descontinuação: não construa dependência crítica nela.
  • LGPD: tratar dados de contato exige política clara de privacidade e base legal.
✓
Recomendação de partida: Fase 1 + Fase 2 já entregam um Radar com sinais verificados de cadastral + notícias + vagas (as fontes mais baratas e confiáveis), com a IA do Claude fazendo intenção e abordagem. LinkedIn entra na Fase 3 quando o ROI justificar.
